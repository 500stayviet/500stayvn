# Phase 4: 스테이징·글로벌 출시 전 체크리스트

**상용 진입 전 문서·운영 클로저(법무·Sentry·웹훅·게이트):** [pre-launch-closure.md](./pre-launch-closure.md)

프로덕션 또는 스테이징 배포 직전에 UI·운영·알림 관점에서 한 번 더 확인한다.  
웹 앱 루트: **`frontend/`** (Next.js).

## UI · 다국어 (앱 내)

| 항목 | 확인 내용 | 상태 |
|------|-----------|------|
| 언어 5종 | KO / VI / EN / JA / ZH 선택 시 주요 화면(탑바, 매물, 예약, KYC 등)에 **다른 언어가 섞이지 않음** | ☐ |
| 탑바 | 알림 패널 빈 상태 문구·미읽음 수·접근성 라벨이 `getUIText` 경로로 표시 | ☐ |
| 주소 모달 | 검색·지도·버튼·에러 `alert`가 동일 5개국어 사전을 참조 | ☐ |

## 이메일 및 알림 다국어화

- **원칙:** 향후 **Nodemailer**, **Resend**, 기타 SMTP/API 기반 **이메일 발송 로직**을 구현할 때는, 반드시 앱의 **i18n 설정**(`frontend/src/utils/i18n.ts`, `i18nListing.ts` 및 병합 규칙)을 참조한다.
- **요구사항:** 수신자(또는 발송 시점의 세션)의 **선택 언어**(`SupportedLanguage`: `ko` | `vi` | `en` | `ja` | `zh`)에 맞춰, **동일 의미의 5개 국어 이메일 템플릿**(제목·본문·푸터·법적 고지 등)을 적용한다. 한 언어만 하드코딩하거나 영어만 보내지 않는다.
- **푸시·인앱 알림:** 서버에서 생성하는 문구가 있다면, 위와 동일하게 **언어별 템플릿** 또는 **메시지 키 + 클라이언트 번역** 전략을 문서화한다.
- **체크:** 스테이징에서 언어별 테스트 계정으로 **예약 확정·비밀번호 재설정·마케팅(해당 시)** 등 트리거 메일 1건 이상씩 수신 확인 | ☐
